## 1. 系统架构设计

### 1.1 整体架构概览

系统采用**前后端分离**架构，前端基于 Next.js 构建，后端基于 FastAPI 构建，通过 RESTful API 进行通信。

```
┌─────────────────────────────────────────────────────────────────┐
│                        客户端 (Browser)                          │
└──────────────────────────────┬──────────────────────────────────┘
                               │ HTTP/HTTPS
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Nginx (反向代理)                              │
│              路由分发: / → 前端, /api/ → 后端                      │
└───────────────────────────────┬─────────────────────────────────┘
                    ┌───────────┴───────────┐
                    ▼                       ▼
┌───────────────────────┐     ┌───────────────────────────────────┐
│   Frontend (Next.js)  │     │        Backend (FastAPI)           │
│   React 19 + TS       │     │                                    │
│   App Router          │     │  ┌─────────────────────────────┐   │
│   Tailwind CSS 4      │     │  │       API 路由层             │   │
└───────────────────────┘     │  │  /api/v1/auth               │   │
                              │  │  /api/v1/collections        │   │
                              │  │  /api/v1/documents          │   │
                              │  │  /api/v1/chat               │   │
                              │  │  /api/v1/search             │   │
                              │  └─────────────────────────────┘   │
                              │  ┌─────────────────────────────┐   │
                              │  │      服务层 (Services)       │   │
                              │  │  ChatService                │   │
                              │  │  DocumentService            │   │
                              │  └─────────────────────────────┘   │
                              │  ┌─────────────────────────────┐   │
                              │  │     RAG 引擎 (核心)          │   │
                              │  │  Reader → Chunker →         │   │
                              │  │  Embedder → Retriever →     │   │
                              │  │  Reranker → Synthesizer     │   │
                              │  └─────────────────────────────┘   │
                              └───────────┬───────────────────────┘
                                          │
          ┌───────────────┬───────────────┼───────────────┬──────────────┐
          ▼               ▼               ▼               ▼              ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  PostgreSQL  │ │    Qdrant    │ │    Redis     │ │    MinIO     │ │  AI Provider │
│  (元数据)     │ │  (向量存储)   │ │  (缓存/队列)  │ │  (文件存储)   │ │  (LLM/Emb)  │
└──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘
```

### 1.2 核心架构决策

| 决策点 | 选择方案 | 理由 |
|--------|---------|------|
| 前端框架 | Next.js 16 (App Router) | SSR/SSG 支持、文件系统路由、React Server Components |
| 后端框架 | FastAPI | 异步支持、自动 API 文档生成、类型安全 (Pydantic) |
| ORM | SQLAlchemy 2.0 (async) | 成熟稳定、异步支持、迁移工具完善 |
| 向量数据库 | Qdrant | 高性能 HNSW 索引、支持本地/远程模式、丰富过滤能力 |
| RAG 框架 | LlamaIndex | 模块化设计、多种嵌入/LLM 后端支持 |
| AI 模型 | OpenAI / Ollama | 云端灵活性 + 本地部署选项 |
| 任务队列 | Celery + Redis | 成熟的分布式任务队列，支持异步文档处理 |
| 对象存储 | MinIO | S3 兼容、轻量级、易于部署 |

### 1.3 RAG 引擎架构

RAG 引擎是本系统的核心模块，采用管道式设计：

```
┌───────────────────────────────────────────────────────────────────┐
│                        RAG Engine Pipeline                        │
├───────────────────────────────────────────────────────────────────┤
│                                                                    │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────────┐   │
│  │ 1.Reader │ → │2.Chunker │ → │3.Embedder│ → │ 4. Qdrant    │   │
│  │ 文档读取  │   │ 文本分块  │   │ 向量嵌入  │   │   向量写入    │   │
│  └──────────┘   └──────────┘   └──────────┘   └──────────────┘   │
│                                                                    │
│  查询流程:                                                         │
│  ┌──────────┐   ┌──────────────┐   ┌──────────┐   ┌───────────┐  │
│  │3.Embedder│ → │5. Retriever  │ → │6.Reranker│ → │7.Synthesizer│ │
│  │ 查询向量化 │   │ 混合检索(RRF)│   │ 结果重排序 │   │ LLM 答案合成 │  │
│  └──────────┘   └──────────────┘   └──────────┘   └───────────┘  │
│                                                                    │
└───────────────────────────────────────────────────────────────────┘
```

**管道各阶段说明：**

| 阶段 | 模块 | 文件 | 功能 |
|------|------|------|------|
| **文档读取** | `reader.py` | `app/rag/reader.py` | 支持 PDF/DOCX/PPTX/XLSX/TXT/MD/HTML/CSV 共 9 种格式 |
| **文本分块** | `chunker.py` | `app/rag/chunker.py` | 4 种策略：递归分块、Markdown 分块、语义分块、父子分块 |
| **向量嵌入** | `embedder.py` | `app/rag/embedder.py` | 支持 OpenAI (`text-embedding-3-small`) 和 Ollama (`bge-m3`) |
| **混合检索** | `retriever.py` | `app/rag/retriever.py` | 向量语义检索 + BM25 关键词检索 + RRF 融合算法 |
| **结果重排序** | `reranker.py` | `app/rag/reranker.py` | Jina Reranker API → Cross-encoder 降级策略 |
| **答案合成** | `synthesizer.py` | `app/rag/synthesizer.py` | 基于 Prompt 模板的 LLM 答案生成，支持多轮对话上下文 |
| **核心编排** | `engine.py` | `app/rag/engine.py` | 统一入口，协调所有组件完成索引和查询流程 |

## 2. 数据库设计

### 2.1 存储职责划分

| 存储系统 | 用途 | 数据内容 |
|---------|------|---------|
| **PostgreSQL** | 关系数据 | 用户、知识库集合元数据、文档元数据、对话历史、消息记录 |
| **Qdrant** | 向量数据 | 文档块的文本向量、Payload 元数据（文件名、路径、块索引等） |
| **Redis** | 缓存/队列 | Celery 任务队列、BM25 索引缓存、会话数据 |
| **MinIO** | 文件存储 | 文档原始文件（PDF、DOCX、PPTX 等） |

#### 2.3.6 Qdrant 向量数据结构

每个知识库对应一个 Qdrant Collection（命名: `kb_{collection_name}`），每个文档块存储为带向量的 Point：

```json
{
  "id": 1234567890,  // hash(file_path + chunk_index) % 2^63
  "vector": [0.012, -0.034, ...],  // 1536维 (OpenAI) 或 1024维 (BGE-M3)
  "payload": {
    "text": "文档块的文本内容...",
    "chunk_index": 0,
    "filename": "报告.pdf",
    "file_path": "col_xxx/uuid.pdf",
    "file_type": "pdf",
    "collection_id": "xxx",
    "document_id": "xxx"
  }
}
```

**Qdrant Collection 配置：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `vector.size` | 1536 | 向量维度，根据嵌入模型自动适配 |
| `vector.distance` | Cosine | 相似度度量方式 |
| `hnsw_config.m` | 16 | HNSW 图中每个节点的连接数 |
| `hnsw_config.ef_construct` | 100 | 构建时的搜索宽度 |
| `optimizers.default_segment_number` | 2 | 默认分段数 |

### 3.5 问答对话

**Base Path:** `/api/v1/chat`

#### 3.5.1 知识库问答

```
POST /api/v1/chat
Authorization: Bearer {token}

Request:
{
  "query": "什么是 RAG 架构？",
  "collection_id": "col-uuid-...",
  "conversation_id": "conv-uuid-...",   // 可选，传入则继续对话
  "top_k": 5,                            // 可选，默认5
  "use_reranker": true                   // 可选，默认true
}

Response (200 OK):
{
  "answer": "RAG（检索增强生成）是一种将信息检索与文本生成相结合的...",
  "sources": [
    {
      "index": 0,
      "source": "AI技术白皮书.pdf",
      "text": "RAG (Retrieval-Augmented Generation) 是...",
      "score": 0.9521
    }
  ],
  "conversation_id": "conv-uuid-..."
}
```

**问答流程：**
1. 验证用户身份和知识库权限
2. 获取/创建对话记录
3. 加载历史对话上下文（最近 20 条消息）
4. 执行 RAG 查询：混合检索 → 重排序 → LLM 合成
5. 保存用户问题和助手回答到消息表
6. 返回答案和引用来源

#### 4.5.2 RRF 融合算法

RRF (Reciprocal Rank Fusion) 是信息检索领域公认的融合算法，公式为：

\[
\text{RRFscore}(d) = \sum_{r \in R} \frac{1}{k + \text{rank}_r(d)}
\]

其中 \(k=60\) 是平滑常数（防止除零），\(R\) 是参与融合的检索结果集（向量检索 + BM25）。

**融合优势：**
- **互补性：** 向量检索擅长语义匹配，BM25 擅长精确关键词匹配
- **鲁棒性：** RRF 无需原始分数归一化，对分数分布不敏感
- **可解释性：** 融合结果标记来源（`vector`、`bm25`、`vector+bm25`）

#### 4.5.3 重排序策略

系统采用三级降级策略：

```
优先级 1: Jina Reranker API (云服务)
  ├─ 需要 JINA_API_KEY
  ├─ 模型: jina-reranker-v2-base-multilingual
  ├─ 异步 HTTP 调用，轻量级
  └─ 失败时降级 ↓

优先级 2: Cross-encoder 本地模型
  ├─ 需要 pip install transformers torch
  ├─ 模型: BAAI/bge-reranker-v2-m3
  ├─ 本地推理，延迟较高但成本低
  └─ 失败时降级 ↓

优先级 3: 原始排序截断
  └─ 直接按 RRF 分数取 top_k，无额外计算
```

## 7. 安全设计考虑

### 7.1 认证安全

| 措施 | 实现方式 |
|------|---------|
| **密码哈希** | bcrypt 算法，不可逆存储 ([auth/__init__.py](file:///Users/chenliang/Documents/code/知识库/backend/app/auth/__init__.py)) |
| **JWT 签名** | HS256 算法，SECRET_KEY 签名 |
| **Token 有效期** | 默认 7 天，可自定义 |
| **用户状态检查** | 每次 API 请求验证 `is_active` 状态 |
| **唯一性检查** | 用户名和邮箱全局唯一，防止重复注册 |

### 7.2 授权安全

| 措施 | 实现方式 |
|------|---------|
| **角色权限** | user/admin 两种角色，管理员拥有更高权限 |
| **资源隔离** | 知识库通过 `owner_id` 关联用户，支持按用户过滤 |
| **依赖注入** | FastAPI Depends 实现声明式授权检查 |

### 7.3 API 安全

| 措施 | 实现方式 |
|------|---------|
| **CORS 控制** | 开发环境允许所有源 (`*`)，生产环境应限制为特定域名 |
| **输入验证** | Pydantic 模型自动验证请求参数类型和格式 |
| **文件上传安全** | 文件类型白名单校验、文件大小限制、空文件检查 |
| **SQL 注入防护** | SQLAlchemy ORM 参数化查询 |
| **错误处理** | 不返回内部错误详情到客户端 |

### 7.4 基础设施安全

| 措施 | 实现方式 |
|------|---------|
| **网络隔离** | Docker 桥接网络 (`kb-network`) 隔离内外网络，仅暴露必要端口 |
| **最小权限** | 各服务使用独立凭据（数据库、MinIO 等） |
| **数据持久化** | 所有数据通过 Docker volumes 持久化存储 |
| **密钥管理** | 敏感信息（API Key、SECRET_KEY）通过环境变量注入 |
| **HTTPS** | 生产环境推荐通过 Nginx 配置 SSL/TLS 终止 |

### 7.5 安全建议（生产部署）

1. **更换默认凭据：** 将 docker-compose 中的默认密码（`kbuser/kbpass`、`kbadmin/kbadmin123`）更改为强密码
2. **配置 HTTPS：** 在 Nginx 中启用 SSL/TLS，使用 Let's Encrypt 免费证书
3. **限制 CORS：** 将 `allow_origins` 从 `["*"]` 改为实际前端域名
4. **生产密钥：** 生成随机 SECRET_KEY（`openssl rand -hex 32`）
5. **API 限流：** 考虑添加速率限制中间件（如 slowapi）
6. **审计日志：** 记录关键操作日志（用户登录、文档操作等）
7. **定期备份：** 配置 PostgreSQL 和 Qdrant 数据的定期备份策略
8. **依赖更新：** 定期更新 Python/Node.js 依赖，修复已知漏洞

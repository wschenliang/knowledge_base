## 1. 系统架构设计

### 1.1 整体架构

系统采用**前后端分离**的微服务架构，通过 Docker Compose 进行容器编排。整体分为六个层次：

```
┌─────────────────────────────────────────────────────────────┐
│                     用户层 (Browser)                         │
│               Next.js 前端 (SSR, App Router)                 │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP/HTTPS
┌──────────────────────────▼──────────────────────────────────┐
│                   网关层 (Nginx)                              │
│             反向代理 / 负载均衡 / SSL 终端                    │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                   应用层 (FastAPI)                            │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌───────────────────┐  │
│  │ 认证服务 │ │ 文档服务 │ │ 问答服务 │ │ Celery Worker    │  │
│  │ JWT +   │ │ 上传/   │ │ Chat +  │ │ 异步文档索引      │  │
│  │ bcrypt  │ │ 管理    │ │ Search  │ │ (Redis 任务队列)  │  │
│  └─────────┘ └─────────┘ └─────────┘ └───────────────────┘  │
└──────┬────────────┬────────────┬──────────────┬─────────────┘
       │            │            │              │
┌──────▼────┐ ┌─────▼─────┐ ┌───▼────┐ ┌──────▼──────────┐
│ PostgreSQL│ │  Qdrant   │ │ Redis  │ │     MinIO       │
│ (元数据)  │ │ (向量库)  │ │(缓存/  │ │   (对象存储)    │
│           │ │           │ │ 消息)  │ │                  │
└───────────┘ └───────────┘ └────────┘ └─────────────────┘
```

### 1.2 组件交互说明

| 数据流 | 路径 | 说明 |
|--------|------|------|
| 用户请求 | Browser → Nginx → FastAPI | 前端通过 API 调用后端服务 |
| 文档上传 | FastAPI → MinIO → PostgreSQL → Qdrant | 文件存储 → 元数据 → 向量化索引 |
| 问答流程 | FastAPI → Qdrant (检索) → LLM (生成) → PostgreSQL (保存) | 混合检索 → RRF 融合 → 重排序 → 答案生成 |
| 异步任务 | FastAPI → Redis → Celery Worker → Qdrant | 文档索引通过 Celery 异步处理 |
| 监控采集 | Prometheus → FastAPI (/metrics) | 拉取指标数据 → Grafana 展示 |

### 1.3 RAG 引擎架构

RAG 引擎是整个系统的核心，采用**管道式架构**：

```
Input Query
    │
    ▼
┌─────────────┐
│ Embedding    │  ← EmbeddingProvider (OpenAI / Ollama)
│ Provider     │     查询向量化
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ Hybrid       │  ← HybridRetriever
│ Retriever    │     向量检索 + BM25 + RRF 融合
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ Reranker     │  ← Cross-encoder / Jina Reranker API
│              │     结果重排序 (精排)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ Synthesizer  │  ← LLM (OpenAI GPT-4 / Ollama Qwen)
│              │     答案合成 + 引用
└──────┬──────┘
       │
       ▼
Output Answer + Sources
```

#### 混合检索算法

系统实现了**向量检索 + BM25 关键词检索 + RRF 倒数排名融合**：

- **向量检索 (Dense)**：通过 Qdrant 的 HNSW 索引进行语义相似度搜索，返回 Top-K * 4 个候选结果
- **BM25 检索 (Sparse)**：基于 `rank-bm25` 库的关键词检索，从 Qdrant 中加载全部文档建立倒排索引
- **RRF 融合**：`score(d) = Σ(1 / (k + rank_i(d)))`，融合向量和 BM25 排名，k=60

### 2.2 向量数据库（Qdrant）

每个知识库集合对应一个 Qdrant Collection，命名规则为 `kb_{collection_name}`（前缀通过 `QDRANT_COLLECTION_PREFIX` 配置）。

#### Point 结构

```python
PointStruct(
    id: int        # hash(文件路径:块索引) % 2^63
    vector: list[float]  # 1536维 (OpenAI) 或 1024维 (bge-m3)
    payload: {
        "text": str,           # 文本块内容
        "chunk_index": int,    # 块索引
        "filename": str,       # 源文件名
        "file_path": str,      # 文件路径
        "file_type": str,      # 文件类型
        "collection_id": str,  # 所属集合ID
        "document_id": str,    # 文档ID
        # ... 可扩展元数据
    }
)
```

#### Collection 配置

- **向量大小**：1536（OpenAI text-embedding-3-small）或 1024（bge-m3）
- **距离度量**：Cosine
- **索引类型**：HNSW (m=16, ef_construct=100)

### 2.3 对象存储（MinIO）

- **Bucket**: `knowledge-base`
- **存储路径**: `{collection_id}/{uuid}{extension}`
- **用途**: 存储文档原始文件，供后续重新处理使用

## 4. 功能模块设计

### 4.1 认证授权模块

**技术方案**：JWT (HS256) + bcrypt 密码哈希

- **Token 有效期**：7 天
- **Token 内容**：`{ sub: user_id, username, role, exp }`
- **密码策略**：bcrypt 哈希（salt rounds 自动）
- **权限模型**：基于角色（user / admin）
- **中间件**：`get_current_user` 依赖注入，自动验证 JWT 和用户状态

**安全特性**：
- 密码不落盘明文
- Token 验证包含过期检查和签名验证
- 禁用用户自动拦截（`is_active=False`）
- 注册时检查用户名和邮箱唯一性

### 4.2 文档管理模块

**核心流程**：

```
用户上传文档
  │
  ├── 1. 格式验证 (is_supported)
  ├── 2. 上传到 MinIO (或本地存储)
  ├── 3. 创建 Document 记录 (status: pending)
  ├── 4. 文档读取 (DocumentReader)
  │      ├── 自动检测编码和格式
  │      └── 支持: PDF, DOCX, PPTX, XLSX, TXT, MD, CSV, HTML
  ├── 5. 文本分块 (TextChunker)
  │      ├── 策略: recursive / fixed
  │      └── 参数: chunk_size=512, chunk_overlap=128
  ├── 6. 向量嵌入 (EmbeddingProvider)
  │      └── OpenAI text-embedding-3-small 或 Ollama bge-m3
  ├── 7. 写入 Qdrant + 更新 BM25 索引
  └── 8. 更新 Document 状态 (indexed)
```

**文件格式支持**：

| 格式 | 扩展名 | 解析库 |
|------|--------|--------|
| PDF | .pdf | unstructured/pdf, pypdf |
| Word | .docx | python-docx, unstructured |
| PowerPoint | .pptx | python-pptx, unstructured |
| Excel | .xlsx | openpyxl |
| Markdown | .md | 内置 |
| 纯文本 | .txt | 内置 |
| HTML | .html | 内置 |
| JSON | .json | 内置 |
| CSV | .csv | 内置 |

### 4.3 知识库集合管理模块

每个知识库集合具有独立的：
- **Qdrant Collection**（命名空间隔离）
- **BM25 索引**（懒加载，按需建立）
- **文档列表**（元数据追踪）
- **分块策略配置**（可自定义 chunk_size 和 chunk_overlap）

**创建流程**：
1. 验证集合名称唯一性
2. 生成 Qdrant collection 名称：`{prefix}{name.lower().replace(' ', '_')}`
3. 创建数据库记录
4. 首次索引文档时自动创建 Qdrant Collection

### 4.4 问答系统模块

#### 4.4.1 检索流程

```
┌───────────────────────────────────────────────────────┐
│                    用户输入 Query                      │
└──────────────────────────┬────────────────────────────┘
                           │
              ┌────────────▼────────────┐
              │   向量嵌入 (Embedding)   │
              │   1536维向量             │
              └────────────┬────────────┘
                           │
         ┌─────────────────▼─────────────────┐
         │         混合检索 (Hybrid)          │
         │  ┌──────────┐   ┌──────────────┐  │
         │  │ 向量检索  │   │  BM25 检索    │  │
         │  │ (Qdrant) │   │ (rank-bm25)  │  │
         │  └────┬─────┘   └──────┬───────┘  │
         │       └──────┬─────────┘          │
         │              ▼                    │
         │       RRF 倒数排名融合             │
         │       k=60                       │
         └─────────────────┬─────────────────┘
                           │  Top-K × 4
              ┌────────────▼────────────┐
              │    重排序 (Reranker)     │
              │  Cross-encoder / Jina   │
              │  API Reranker           │
              └────────────┬────────────┘
                           │  Top-K (默认5)
              ┌────────────▼────────────┐
              │   答案合成 (Synthesizer) │
              │   LLM + 提示词模板       │
              │   + 检索上下文           │
              └────────────┬────────────┘
                           │
              ┌────────────▼────────────┐
              │   返回: Answer + Sources │
              └─────────────────────────┘
```

#### 4.4.2 重排序策略

系统检测`JINA_API_KEY`环境变量，自动选择合适的重排序模式：
- **有 JINA_API_KEY**: 使用 Jina Reranker API（云端，支持多语言）
- **无 JINA_API_KEY**: 尝试加载本地 Cross-encoder 模型（BAAI/bge-reranker-v2-m3）

#### 4.4.3 答案合成

- **LLM 选择**：有 `OPENAI_API_KEY` 使用 GPT-4o-mini，否则使用 Ollama qwen2.5:7b
- **提示词模板**：包含系统角色定义、引用规范、知识范围约束

### 4.5 异步任务模块

使用 Celery + Redis 实现异步任务处理：

```
FastAPI 上传文档
     │
     ├── 保存到 MinIO + PostgreSQL
     └── 提交 Celery 任务
             │
        Redis (Broker)
             │
        Celery Worker
             │
        ├── 文档读取
        ├── 文本分块
        ├── 向量嵌入
        ├── 写入 Qdrant
        └── 更新文档状态
```

**任务配置**：
- 最大重试：3 次
- 重试间隔：60 秒
- 任务超时：30 分钟（硬限制）/ 25 分钟（软限制）

## 7. 安全设计考虑

### 7.1 认证安全

- **JWT 签名**：HS256 算法，SECRET_KEY 至少 32 字符随机字符串
- **密码存储**：bcrypt 自适应哈希，自动 salt
- **Token 过期**：7 天自动过期
- **Token 验证**：每次请求验证签名、过期时间、用户状态

### 7.2 API 安全

- **CORS 配置**：开发环境允许所有来源；生产环境应限制具体域名
- **输入验证**：Pydantic 自动验证请求体；文件上传检查格式和大小
- **SQL 注入防护**：SQLAlchemy 参数化查询，ORM 自动转义
- **依赖注入**：敏感操作通过 `get_current_user` 验证身份

### 7.3 数据安全

- **数据库隔离**：PostgreSQL 独立用户和数据库
- **密码隔离**：MinIO 和 PostgreSQL 使用独立凭据
- **数据持久化**：所有数据存储在 Docker named volumes，防止容器销毁丢失
- **文件安全**：上传文件通过对象存储隔离（MinIO Bucket），不直接暴露在文件系统

### 7.4 网络安全

- **Docker 网络隔离**：服务间通过 `kb-network` 内部通信，仅暴露必要端口
- **生产环境端口**：仅 Nginx (80/443) 向外暴露，其他服务通过内部网络访问
- **SSL**：生产环境建议在 Nginx 配置 SSL 证书

### 7.5 运维安全

- **非 root 运行**：Dockerfile 中应用以非特权用户运行
- **健康检查**：所有关键服务配置 healthcheck，自动重启异常容器
- **日志管理**：JSON file logging driver，自动轮转避免磁盘写满
- **资源限制**：生产环境配置 CPU/内存限制防止资源耗尽

### 7.6 安全隐患清单

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 默认凭据 | MinIO、PostgreSQL 使用默认密码 | 生产环境通过 .env 覆盖 |
| SECRET_KEY 硬编码 | 开发环境使用固定密钥 | 生产环境注入随机密钥 |
| CORS 全开 | `allow_origins=["*"]` | 生产环境限制为前端域名 |
| DEBUG 模式 | 开发环境开启 DEBUG | 生产环境 `DEBUG=false` |
| API Key 泄露 | .env 文件可能被提交 | 添加到 .gitignore |

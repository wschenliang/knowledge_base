RAG 技术入门与实践
===================

一、什么是 RAG
--------------

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大型语言模型（LLM）相结合的技术架构。它的核心思想是在 LLM 生成回答之前，先从外部知识库中检索出相关文档片段，然后将这些检索结果作为上下文提供给 LLM，使其能够基于真实、准确的信息生成答案。

二、为什么需要 RAG
-----------------

2.1 解决 LLM 的局限性

- 知识截止问题：LLM 的训练数据有截止日期，无法获取最新信息
- 幻觉问题：LLM 可能编造不存在的"事实"
- 专业领域知识不足：通用 LLM 缺乏企业内部的专业知识
- 可追溯性差：LLM 的回答无法直接关联到具体的数据来源

2.2 RAG 的优势

- 时效性强：文档更新后立即影响回答质量
- 可解释性高：回答附带了引用来源
- 成本可控：不需要重新训练模型
- 支持私有数据：企业敏感数据无需出域
- 效果稳定：检索到的文档质量决定了回答的上限

三、RAG 架构的核心组件
--------------------

3.1 文档加载器（Document Loader）

负责从各种来源加载文档，支持多种格式：
- PDF 文档解析
- Word 文档解析
- Excel 表格解析
- 网页内容抓取
- 数据库记录提取

3.2 文本分块器（Text Chunker）

将长文档切分成适合检索的短片段，常见策略包括：
- 递归分块：按段落、句子层级递归切分
- 语义分块：基于语义完整性进行切分
- 父子分块：保留大块上下文的同时生成小块供检索

3.3 嵌入模型（Embedding Model）

将文本转换为向量表示：
- OpenAI text-embedding-3-small：1536 维，适合通用场景
- BGE-M3：1024 维，多语言支持好
- 嵌入向量之间的余弦相似度衡量文本语义相关性

3.4 向量数据库（Vector Database）

存储和检索向量数据：
- Qdrant：高性能 HNSW 索引
- Milvus：分布式向量数据库
- Pinecone：云原生向量服务
- Chroma：轻量级嵌入式向量数据库

3.5 检索器（Retriever）

实现混合检索策略：
- 向量检索：基于语义相似度
- BM25 关键词检索：基于词频统计
- RRF 融合：将两种检索结果的排序进行融合

3.6 重排序器（Reranker）

对初步检索结果进行精排：
- Cross-encoder 模型：对查询-文档对进行深度匹配评分
- Jina Reranker API：云端多语言重排序服务
- 可以显著提升 top-k 结果的准确率

3.7 答案合成器（Synthesizer）

将检索结果和用户问题喂给 LLM 生成最终答案：
- 构建包含上下文信息的提示词
- LLM 基于上下文生成准确回答
- 可配置不同的提示词模板

四、混合检索详解
--------------

混合检索是 RAG 系统性能的关键。

向量检索的优点：
- 理解语义相似性（"苹果"和"iPhone"语义相近）
- 对同义词和近义词效果好
- 不受具体措辞限制

BM25 检索的优点：
- 精确匹配关键词（如产品型号、人名）
- 对长尾关键词效果好
- 计算速度快，无需 GPU

RRF（Reciprocal Rank Fusion）：
- 将两种检索结果的排名加权融合
- 公式：RRF(d) = sum(1 / (k + rank_i(d)))
- 其中 k=60 为平滑常数
- 有效结合两种检索方式的优势

五、企业级 RAG 系统设计要点
-------------------------

5.1 数据安全

- 支持本地部署，数据不出域
- 支持私有化嵌入模型和 LLM
- 基于角色的访问控制（RBAC）

5.2 可扩展性

- 支持多知识库，按业务场景隔离数据
- 异步文档处理，支持大批量文档导入
- 支持多种嵌入模型和 LLM 后端

5.3 可观测性

- 检索质量的指标监控
- 回答准确性的评估体系
- 用户反馈闭环

六、应用场景
----------

1. 企业内部知识库问答
2. 产品文档智能客服
3. 法律文档检索与咨询
4. 医疗文献辅助诊断
5. 教育领域智能辅导
6. 技术支持问题自动回复
